## やったこと

### AWS SAM local の仕組みを深掘りした

`sam local` がローカル環境でLambdaやAPI Gatewayをどのように再現しているのか、技術的な裏側を調べた。

---

#### 1. Lambda環境の構築（Dockerの使用）

Lambda の実行環境は、Docker コンテナを利用して**本番環境とほぼ同一の構成**をシミュレートしている。

**AWS 公式イメージの利用**

AWS は実際のLambda環境（Amazon Linux、各種言語ランタイム、システムライブラリなど）をパッケージ化した公式Dockerイメージ（`public.ecr.aws/sam/emulation-xxx` など）を提供している。`sam local` 実行時、SAM CLIは `template.yaml` の `Runtime` 指定を確認し、対応するイメージを自動的に取得（Pull）する。

**ローカルコードのボリュームマウント**

コンテナ起動時、ローカルPC上のソースコード（通常は `sam build` 後の `.aws-sam/build/` 内）をDockerの**ボリュームマウント機能**を使ってコンテナ内の `/var/task` などに直接共有する。これにより、コンテナを毎回作り直さずとも最新のコードを実行できる。

**Lambda Runtime API Emulator (RIE)**

コンテナ内では **Lambda Runtime API Emulator (RIE)** という軽量な実行ファイルが動作している。これがLambdaサービス本体の代わりとなり、外部からのリクエスト（JSON）を受け取って関数を実行し、レスポンスを返す役割を担う。

---

#### 2. API Gateway環境の構築（SAM CLIの機能）

API GatewayについてはDockerを使わず、**SAM CLI自体がプロキシサーバーとして動作**する。

**ローカルHTTPサーバーの起動**

`sam local start-api` を実行すると、SAM CLIがローカルPC上で軽量なHTTPサーバー（デフォルト: `localhost:3000`）を立ち上げる。

**HTTPリクエストの変換（イベント化）**

ブラウザなどからHTTPリクエストを受け取ると、SAM CLIはそれを**「API Gatewayプロキシイベント」**（Lambdaが受け取るJSONデータ）に変換する。URLパス、HTTPメソッド、ヘッダー、クエリパラメータ、Bodyなどを抽出し、特定のJSON構造に整形する。

**コンテナへの橋渡し**

整形されたJSONイベントは、Lambda Dockerコンテナへ送信される。Lambdaから返ってきたJSONレスポンスを、SAM CLIが再び通常のHTTPレスポンスに逆変換してクライアントへ返す。

---

#### 3. リクエスト処理の全体フロー

`sam local start-api` にリクエストが飛んできた時の内部処理フローは以下の通り。

1. **[SAM CLI]** ユーザーからのHTTPリクエストを `localhost:3000` で受信
2. **[SAM CLI]** リクエスト内容をAPI Gateway形式の**JSONイベントに変換**
3. **[Docker]** 指定ランタイムの**コンテナを起動**（または待機中のものを利用）し、ローカルコードをマウント
4. **[Docker]** コンテナ内の**RIE**がJSONイベントを受け取り、Lambda関数コードを実行
5. **[SAM CLI]** コンテナからの実行結果（JSON）を受信
6. **[SAM CLI]** 受信データを**HTTPレスポンスに変換**してユーザーに返却
7. **[Docker]** 設定に応じてコンテナを停止、または次回の呼び出しのために維持

**「API構造の模倣はSAM CLI」**、**「実行環境の模倣はDocker」**と役割を分担することで、本番に近い環境をローカルに構築している。
