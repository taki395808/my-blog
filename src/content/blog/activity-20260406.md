### バケッティングの実例を調べた

テキストの例では年月日でデータが分散されていたため、「これはパーティショニングと何が違うのか？」という疑問が生じた。調べた結果、**バケッティングはパーティショニングと目的が異なる**ことが分かった。

---

#### パーティショニングとバケッティングの違い

| 観点 | パーティショニング | バケッティング |
|------|------------------|--------------|
| 分割の基準 | カラム値の等値（`year=2024` など） | カラム値のハッシュ値 |
| 向いているカラム | **低カーディナリティ**（種類が少ない） | **高カーディナリティ**（種類が多い）|
| S3上の構造 | ディレクトリ階層（`year=2024/month=01/`）| フラットなファイル群 |
| スキャン削減 | WHERE句でパーティションごとスキップ | 特定バケットのみスキャン |
| JOIN / GROUP BY 性能 | 中程度 | 優秀（同じキーが同じファイルに集まる）|
| 過剰使用のリスク | 小ファイルが大量生成される | バケット数が多すぎると計画オーバーヘッド |

年月日のような種類が少ない列はパーティショニング向き。ユーザーIDや商品IDのように種類が膨大な列はバケッティング向き、というのが整理のポイント。

---

#### 年月日以外のバケッティング実例

**ユースケース 1 — ユーザーID（ECサイト行動ログ）**

数百万〜数億規模のユーザーIDをパーティション分割すると、小ファイルが爆発的に増加する。バケッティングなら指定した数のファイルに収めつつ、特定ユーザーのクエリは1バケットのみスキャンできる。

```sql
CREATE TABLE user_events_bucketed
WITH (
    format = 'PARQUET',
    external_location = 's3://my-datalake/user_events_bucketed/',
    partitioned_by = ARRAY['event_type'],   -- 低カーディナリティ → パーティション
    bucketed_by = ARRAY['user_id'],         -- 高カーディナリティ → バケット
    bucket_count = 64
)
AS SELECT user_id, session_id, timestamp, page, event_type
FROM raw_events;
```

**ユースケース 2 — デバイスID（IoTセンサーデータ）**

```sql
CREATE TABLE sensor_data_bucketed
WITH (
    format = 'PARQUET',
    partitioned_by = ARRAY['region'],
    bucketed_by = ARRAY['device_id'],
    bucket_count = 128
)
AS SELECT device_id, sensor_type, value, ts, region
FROM raw_sensor_data;
```

**ユースケース 3 — 商品ID（注文×在庫のJOIN高速化）**

注文テーブルと在庫テーブルを `product_id` でJOINするとき、両テーブルを同じ `product_id` でバケッティングしておくと、全ノード間のデータ転送（Shuffle）が不要になり大幅に高速化される。

**ユースケース 4 — 気象観測所ID（パーティション＋バケットの複合戦略）**

```
パーティション: report_type（DAILY / MONTHLY）  ← 低カーディナリティ
バケット: station_id（数万局）                  ← 高カーディナリティ
```

クエリは `report_type` パーティションを絞り込んだ後、対象の `station_id` が入るバケットのみスキャンする、という2段階の絞り込みが可能になる。

#### AWS Glue での注意点

**DynamicFrame はバケッティング非対応**。`writeDynamicFrame` ではなく、Spark DataFrame の `write.bucketBy()` を使う必要がある。

```python
# AWS Glue PySpark での実装例
df.write \
    .format("parquet") \
    .partitionBy("event_type") \
    .bucketBy(64, "user_id") \
    .sortBy("user_id") \
    .saveAsTable("user_events_bucketed")
```

### Amazon QuickSight のビジュアライゼーションを整理した

QuickSightで使えるビジュアルタイプは30種類以上ある。代表的なものを整理した。

| ビジュアルタイプ | 用途・適したシーン |
|---------------|----------------|
| **AutoGraph** | データに最適なグラフを自動選択（初期分析に便利）|
| **棒グラフ** | カテゴリ別の値比較（売上ランキング、地域別件数）|
| **折れ線グラフ** | 時系列トレンド（日別アクセス数、月次売上推移）|
| **複合グラフ** | 棒グラフ＋折れ線の重ね表示（売上と前年比率の同時表示）|
| **散布図** | 2変数の相関（広告費 vs 売上、遅延時間 vs エラー率）|
| **ヒートマップ** | 2次元のデータ密度（曜日×時間帯のアクセス分布）|
| **ツリーマップ** | 階層構造の相対的サイズ（カテゴリ別売上シェア）|
| **KPI** | 重要指標と目標との比較（当月売上 vs 目標）|
| **ピボットテーブル** | 多次元クロス集計（商品×地域×期間の売上）|
| **箱ひげ図** | 分布の外れ値検出（処理時間の中央値・外れ値分析）|
| **サンキーダイアグラム** | フロー・流量の可視化（ユーザー遷移、データ流通経路）|
| **地図・地理空間グラフ** | 地理的分布（国・都道府県別の件数）|

**データエンジニアリング文脈での典型的なアーキテクチャ:**

```
生データ(S3 CSV/JSON)
  → Glue Crawler（スキーマ自動検出）
  → Glue ETL（Parquet変換・パーティション付与）
  → Athena（SQLクエリ）
  → QuickSight SPICE（高速インメモリキャッシュ）
  → ダッシュボード
```

2024年からは「Amazon Q in QuickSight」として生成AI機能が追加されており、自然言語でダッシュボードへの質問や要約生成が可能になっている。

### Amazon Athena CTAS を調べた

テキストに「CATS」という表記があったが、正確には **CTAS（CREATE TABLE AS SELECT）** のこと。

---

#### CTASとは

`SELECT` の結果から新しいテーブルをAthenaで直接作成する構文。

- データは指定したS3パスに自動保存される
- テーブル定義はAWS Glue Data Catalogに自動登録される
- 出力フォーマット（Parquet/ORC/Avro/JSON/CSV）と圧縮方式を指定できる

---

#### 主要な使い方

**1. フォーマット変換（CSV → Parquet）**

```sql
CREATE TABLE logs_parquet
WITH (
    format = 'PARQUET',
    write_compression = 'SNAPPY',
    external_location = 's3://my-bucket/logs_parquet/'
)
AS SELECT * FROM logs_csv;
```

**2. フォーマット変換 + パーティション付与**

```sql
CREATE TABLE logs_partitioned
WITH (
    format = 'PARQUET',
    write_compression = 'SNAPPY',
    external_location = 's3://my-bucket/logs_partitioned/',
    partitioned_by = ARRAY['year']
)
AS SELECT
    request_id, status_code, latency_ms, user_id,
    substr(request_date, 1, 4) AS year
FROM logs_csv
WHERE status_code >= 400;
```

**3. パーティション + バケット（複合最適化）**

```sql
CREATE TABLE events_optimized
WITH (
    format = 'PARQUET',
    external_location = 's3://my-bucket/events_optimized/',
    partitioned_by = ARRAY['event_type'],
    bucketed_by = ARRAY['user_id'],
    bucket_count = 64
)
AS SELECT user_id, event_type, timestamp, page
FROM raw_events;
```

**4. Apache Icebergテーブル（タイムトラベル対応）**

```sql
CREATE TABLE events_iceberg
WITH (
    table_type = 'ICEBERG',
    format = 'PARQUET',
    location = 's3://my-bucket/events_iceberg/',
    is_external = false,
    partitioning = ARRAY['month(event_date)']
)
AS SELECT event_id, user_id, event_type, event_date
FROM raw_events;
```

---

#### ETLでの主要活用パターン

| パターン | 内容 | 効果 |
|--------|------|------|
| フォーマット変換 | CSV/JSON → Parquet へ変換 | スキャン量を最大96%削減 |
| パーティション付与 | 非パーティションデータを年月日等で分割 | 不要パーティションをスキップ |
| 圧縮 | Snappy/Zstd で圧縮 | ストレージコスト削減・速度向上 |
| データ絞り込み | WHERE句で必要行・列だけ抽出 | 後続クエリの対象データ削減 |
| Iceberg変換 | 既存テーブルをIcebergに移行 | タイムトラベル・ACID対応 |

**実測パフォーマンス例（AWSブログより）:**

| 操作 | 元のCSV | Parquet変換後 |
|------|---------|-------------|
| クエリ実行時間 | 16.88秒 | 3.79秒（4.4倍高速）|
| スキャンデータ量 | 11.35 GB | 428 MB（96%削減）|

---

#### CTASの制限事項（試験対策メモ）

- INSERT INTO は1回あたり**最大100パーティションまで**（超える場合は複数回実行が必要）
- CTAS で作成したテーブルを `DROP TABLE` しても**S3のデータは削除されない**（外部テーブルの仕様）
- CTAS 自体はトランザクションではない（Icebergを使う場合を除く）
- 書き込み先S3パスは**事前に空である**必要がある
