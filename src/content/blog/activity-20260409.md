### インターネットゲートウェイについて確認

インターネットゲートウェイ（IGW）を一般的な物理ネットワーク機器に例えると、企業ネットワークにおける**「エッジルーター（境界ルーター）」や、オフィス・家庭にある「ブロードバンドルーター」**が最も当てはまる。

#### そのように例えられる主な理由

**1. インターネットとの「出入口」として機能する**

物理ネットワークにおいて、社内ネットワーク（LAN）から外部のインターネット（WAN）へ接続するためには、ネットワークの境界にルーターを配置して通信を外へ逃がす必要がある。AWSにおけるVPC（社内ネットワーク）とインターネットの境界に立ち、通信の関所となるのがインターネットゲートウェイである。

**2. グローバルIPとプライベートIPの橋渡し（NAT的な役割）**

オフィスからインターネットを見る際、通常はルーターが社内のプライベートIPアドレスと、インターネット用のグローバルIPアドレスを変換（NAT/NAPT）している。AWSのインターネットゲートウェイも同様に、VPC内のEC2インスタンスなどに割り当てられた「パブリックIPv4アドレス」と「プライベートIPv4アドレス」の1対1の変換を（ユーザーからは透過的に）行い、インターネットとの通信を成立させている。

---

#### 【対比】VPCエンドポイントを一般的なネットワーク機器に例えると？

VPCエンドポイント（特にGateway型やInterface型）は、**「特定の関連会社や特定サービスと直接通信するために引かれた、専用線（閉域網）の接続用ルーター」**に例えることができる。

| 機器 | 例え | 役割 |
|------|------|------|
| インターネットゲートウェイ（エッジルーター） | 「不特定多数のいる公道」に出るための汎用的なメインゲート | インターネット全般への通信 |
| VPCエンドポイント（専用線の接続口） | 安全な「地下の専用通路」を通る専用ゲート | S3・DynamoDBなど特定AWSサービスへの閉域通信 |

業務システムやデータ基盤を構築する際、外部に公開する必要のないデータ処理サーバーなどがAWSのマネージドサービス（S3など）と通信する場合は、セキュリティの観点から「公道」であるインターネットゲートウェイは通らず、「専用通路」であるVPCエンドポイントを経由させるのが一般的なアーキテクチャとなる。

### CloudWatch Logs Insights について確認

CloudWatch Logs Insightsで実務上よく利用される具体的なクエリ例を確認した。対象となるログ（Lambda、API Gateway、VPCフローログなど）の形式によってフィールド名が変わるため、一般的なユースケースに基づいた例を整理する。

#### 1. エラーログの抽出とカウント

システム内で発生したエラーを素早く特定し、どのエラーがどれだけ発生しているかを集計する。

```
fields @timestamp, @message, @logStream
| filter @message like /(?i)(error|exception|fail)/
| stats count() as ErrorCount by @message
| sort ErrorCount desc
| limit 20
```

`error`・`exception`・`fail`という文字列（大文字・小文字を問わず）を含むログを抽出し、メッセージごとに発生回数をカウントして多い順に並べる。

#### 2. AWS Lambda の実行時間の分析（遅い順）

Lambda関数のパフォーマンス低下を調査するため、実行時間（Duration）が長いものを特定する。

```
fields @timestamp, @logStream, @duration, @billedDuration, @memorySize, @maxMemoryUsed
| filter @type = "REPORT"
| sort @duration desc
| limit 10
```

Lambdaが実行終了時に出力する `REPORT` ログを抽出し、実際の実行時間（`@duration`）が長い順にソートしてトップ10を表示する。

#### 3. API Gateway のステータスコード別リクエスト数

APIの健全性を確認するために、ステータスコード（200番台・400番台・500番台など）の割合を集計する。

```
fields @timestamp, status, httpMethod, resourcePath
| stats count() as RequestCount by status
| sort RequestCount desc
```

アクセスログから `status` フィールドを抽出し、ステータスコードごとにリクエスト数を集計する。エラーレートの急増を検知する際などに役立つ。

#### 4. VPCフローログからの拒否（REJECT）トラフィックの特定

セキュリティグループやネットワークACLでブロックされた通信の送信元IPを特定する。

```
fields @timestamp, srcAddr, dstAddr, dstPort, protocol, action
| filter action = "REJECT"
| stats count() as RejectCount by srcAddr, dstPort
| sort RejectCount desc
| limit 20
```

`action` が `REJECT` となっているログを抽出し、どの送信元IP（`srcAddr`）からどのポート（`dstPort`）へのアクセス試行が多かったかを集計する。

#### 5. 特定の時間枠における特定ユーザーの行動追跡

アプリケーションログなどで、特定のユーザーIDがどのようなアクションを行ったかを時系列で追跡する。

```
fields @timestamp, @message
| filter @message like /user_id_12345/
| sort @timestamp asc
| limit 100
```

指定したユーザーIDが含まれるログを、古いものから順に（`asc`）表示することで、ユーザーの一連の操作フローを確認できる。
